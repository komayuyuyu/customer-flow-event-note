# イベント候補データ形式

イベント情報を更新するときは、候補を `data/candidates.json` に書き、`scripts/evaluate_candidates.py` で影響度・確からしさ・影響時間帯を計算する。

週1回の広めチェックでは、実行日を基準に当日から60日先までを候補確認範囲にする。毎朝チェックでは、当日と直近の更新・中止・延期・放送変更を確認する。

```json
[
  {
    "title": "イベント名",
    "startAt": "2026-07-05T21:00:00+09:00",
    "endAt": "2026-07-05T23:00:00+09:00",
    "broadcast": "地上波・無料配信",
    "category": "スポーツ中継",
    "area": "全国",
    "liveReason": "決勝戦のためリアルタイム視聴の動機が強い",
    "nationalReach": 3,
    "liveUrgency": 3,
    "accessibility": 3,
    "buzz": 3,
    "significance": 3,
    "officialConfirmed": true,
    "sources": [
      {"title": "公式日程", "url": "https://example.com/official"},
      {"title": "報道", "url": "https://example.com/news"}
    ]
  }
]
```

## 追加できる表示用フィールド

- `category`: `スポーツ中継`、`花火`、`祭り`、`パレード`、`イルミネーション`、`セール関連` など。
- `area`: `兵庫`、`大阪`、`京都`、`東京`、`全国` など。
- `trafficReason`: 客足に影響しそうな理由。地域イベントでは交通規制、来場集中、帰宅混雑などを書く。
- `inPersonEvent`: `true` にすると、予測時間帯が `来場・交通混雑`、`イベント開催中`、`終了後の帰宅混雑` になる。
- `impactLevelOverride`: 神戸の勤務先への影響度を手動で調整する。東京など遠方の現地イベントは、テレビ放送・全国的な話題化が見込める場合だけ `大` にする。
- `championship`: ヴィッセル神戸など優勝接近イベント用の詳細。

## 勤務カレンダー由来イベント

勤務カレンダー・シフト表の「イベント関連」欄から入れる店内イベントは `data/store-events.json` に書く。

- 長期イベントは `showEachDay: true` を付け、`startAt` と `endAt` に実際の開始日・終了日を入れる。
- `predictedWindows[0].date` は開始日と合わせる。
- 表示はしたいが通常日扱いで保存したいイベントは `recordLink: false` を付ける。現在の固定対象は `棚卸`。
- `recordLink: false` のイベントは、EVENT SIGNAL と週間予定には表示されるが、集客記録の `relatedEvents` / `eventIds` には入らない。
- `レイアウトUP`、`館店長会`、`施策〆`、`OL店長会`、`HELP ○○` は掲載不要。

```json
{
  "championship": {
    "team": "ヴィッセル神戸",
    "winsToTitle": 2,
    "condition": "ヴィッセルが2連勝し、2位が次節で勝点を落とすと優勝決定",
    "runnerUpCondition": "2位が勝利し、得失点差を6点以上詰めた場合のみ逆転"
  }
}
```

## 0〜3の判定軸

- `nationalReach`: 全国・年代横断の認知度
- `liveUrgency`: その時刻に見る必要性
- `accessibility`: 地上波・無料配信などの見やすさ
- `buzz`: 公開SNS・ニュースでの話題性
- `significance`: 決勝、最終公演、代表戦など一回性の強さ

店舗のお客様層は判定に使わない。

## 表記ルール

- サッカーのワールドカップは `サッカーW杯` と表記する。
- 女子バレーボールのVNLは `VNL女子` と表記する。
- ヴィッセル神戸の優勝接近時は、タイトルの先頭を `ヴィッセル神戸` に統一する。
- 対戦国名は半角英大文字3文字の国コードで表記する（例: `JPN対BRA`、`JPN対THA`）。
- スポーツの進行段階を示すラウンドは、大文字 `R` と数字で表記する（例: `R32`、`R16`）。
- 同じ大会の全イベントへ、完全に同じイベント名を必ず付ける。
- 各行を単独で読んでも、何の競技・大会か分かる状態にする。

## 絶対ルール

- 週1回の広めチェックでは、実行日から60日先までのイベント情報を確認する。
- 既存イベントは過去分を含めて削除せず、イベントID単位で追加・更新する。
- 中止・延期になったイベントも履歴として残し、削除ではなく状態を更新する。
- 営業時間外でも除外しない。
- 夜のイベントは2〜3時間前からの帰宅準備を評価する。
- 深夜・早朝のイベントは、その日の午前〜昼の外出控えを評価する。
- 公式確認と複数情報源を区別して、確からしさを付ける。
- 花火、ルミナリエ、優勝パレード、神戸祭りのような広域人流イベントは、兵庫、大阪、京都、東京を優先して確認する。
- 祭りは、関西以外でも日本全国の誰もが知っている規模のものは収集対象に含める。
- 東京など遠方の現地イベントは、神戸の勤務先への直接影響を過大評価しない。テレビ放送や全国的な話題化で影響が見込める時だけ `影響 大` とする。
- ヴィッセル神戸が優勝争いに入った場合は、次戦以降の試合をイベント候補に入れ、`championship` に「あと何勝」「優勝条件」「2位以下の逆転条件」を書く。
